# t5_dispatch A/B: does `sf` earn its tokens on many-file structural mapping?

The third attempt in this effort, and the first designed squarely inside
`sf`'s *documented* strength rather than at (t4) or outside (t1) its boundary.
The two prior runs are prerequisites for reading this one:

- [`t1_calllog`](./2026-07-02-t1-calllog.md) ‚Äî **invalid.** The task was
  grep-shaped (enumerate which files match a pattern across ~22 files). The
  `sf hook pre` PreToolUse hook only gates a *full* `Read`/`cat` of a big
  source file ‚Äî never `Grep` ‚Äî so on a grep-shaped task the hook never fires
  and `sf` was never invoked in any rep. You can't attribute a cost delta to
  an unused tool.
- [`t4_packagist`](./2026-07-02-t4-packagist-forced.md) ‚Äî **valid, a real
  loss.** Single-file exhaustive comprehension of one ~12KB file. `sf` was
  genuinely used (structural-first, 27 calls, zero full Reads) and still lost
  (+8.6% $, +94% tokens), because reconstructing *almost all* of one file
  symbol-by-symbol costs more round-trips than one `Read`. That is `sf`'s
  documented boundary: drill-down does not pay off when you end up needing a
  single file in full.

`t5_dispatch` targets the regime where the prior private study (`micro.md`'s
`t1_deal`) found `sf` *did* win on dollars: **many files, where you need each
file's *shape* but not its full body.** The task maps the `sf` CLI's own
command-dispatch architecture across **13 files in 12 packages** ‚Äî for each
file, its declared types (with struct fields), its function signatures, and
how the dispatch/telemetry spine wires them together. Crucially, this is
answerable from `sf code <file>`'s one-shot structural summary (which prints
every struct's fields and every signature ‚Äî verified below) *without* opening
function bodies; the plain arm, to enumerate the same struct fields and
signatures reliably, ends up reading the files in full.

**How this differs from both priors, by design:**
- Unlike **t1**, it is *not* grep-shaped. It needs each file's structural
  shape (struct fields, full signatures, the Options‚ÜíRun‚ÜíCollect contract),
  which `grep '^func\|^type'` does not give ‚Äî the plain arm confirmed this by
  doing **16‚Äì24 full `Read`s per rep, near-zero grep-hunting** (verification
  below). The hook has real full-`Read`s to engage.
- Unlike **t4**, no single file is needed in full. The answer lives in
  signatures/types across *many* files, so `sf` uses cheap `sf code` structural
  reads (near-zero full Reads) instead of drilling one file symbol-by-symbol.

**Commands:**

```
# free mechanical proof first (Step 2 below), then:
REPS=1  ARMS=sf    TASKS=t5_dispatch bash run.sh                 # pilot = sf rep 1
REPS=3  ARMS=plain TASKS=t5_dispatch bash run.sh                 # plain reps 1-3
REP_START=2 REPS=2 ARMS=sf TASKS=t5_dispatch bash run.sh         # sf rep 2
REP_START=3 REPS=3 ARMS=sf TASKS=t5_dispatch bash run.sh         # sf rep 3
bash judge.sh
bash aggregate.sh
```

Hook mode was the **production default `nudge`** (not t4's forcing `strict`) ‚Äî
this task is genuinely in `sf`'s wheelhouse, so the structural-first preamble
was expected to route the agent to `sf code` on its own, with the nudge hook
as a backstop. It did (see usage verification).

**Date:** 2026-07-02. **Model:** sonnet (both arms). **REPS:** 3.
**BASE_SHA:** `257718bfc4d6fee74322c24f4c90e8db02c99efa` (sofia-ctx/sofia) ‚Äî
confirmed an ancestor of the `../sofia` clone's HEAD (`35c34ee`); the hook
code is byte-identical at BASE_SHA and HEAD, so the installed binary's gating
matches the frozen target.

## TL;DR

1. **This time `sf` won on dollars ‚Äî a genuine, in-boundary win.** Median cost
   `sf` **$0.754** vs `plain` **$0.926** ‚Äî **‚àí18.6%** ‚Äî while `sf` processed
   **+56% more total tokens**. That is exactly the `micro.md`/`t1_deal`
   mechanism, now reproduced on public code: `sf` shifts volume into cheap
   `cache_read` and *avoids the expensive `cache_creation` of full Reads*
   (`sf` `cache_creation` **‚àí47%** vs plain). The dollar win is real but rests
   on that pricing shift, not on fewer tokens.
2. **`sf` was genuinely exercised, structural-first, across all 13 files.**
   130 `sf code` calls + 1 `sf grep` over the three reps (47/33/50 `sf code`
   per rep), and only **2 full-`Read` *attempts* per rep ‚Äî both denied by the
   nudge hook, after which the agent went structural. Zero full Reads
   completed. The `plain` arm did the opposite: **16‚Äì24 full `Read`s per rep,
   zero `sf`** ‚Äî the expensive path this task was built to expose.
3. **But the win is fragile at N=3 and the stable signal favours plain.**
   `sf`'s three reps *straddle* plain: its two cheaper reps ($0.74/$0.75) beat
   every plain rep, but its most-thorough rep ($1.14, the pilot) was *more*
   expensive than all three plain reps. Token volume (`billed_in`, the
   cache-invariant metric) is **+56% higher** on `sf` every time ‚Äî consistent
   with every prior measurement that `sf` is never "fewer tokens." The dollar
   win is the cache-pricing shift only, and dollars carry cache-warmup noise.
4. **Quality is equal.** All 6 runs passed the judge (empty diffs, as
   required). Scores: `sf` 94/88/92 (median 92), `plain` 98/88/94
   (median 94) ‚Äî parity.

## Step 2 ‚Äî mechanical proof the hook engages on these 13 files (free)

real files in a BASE_SHA worktree, modes `nudge`/`strict`, `minBytes=4096`
(the real defaults). A full `Read` of each in-scope file:

nudge/strict  full Read  internal/common/github/github.go       -> deny  (12593 B)
nudge/strict  full Read  internal/common/worktrees/worktrees.go -> deny  (7098 B)

So 11 of the 13 files (all >4096 B) deny a full `Read`; the two tiny spine
files (main.go, root.go) pass ‚Äî both are small enough to read cheaply either
way. Chunked reads and `Grep` pass through untouched. Also confirmed free:
`sf code <file>` prints **the full struct fields and every signature in one
call** ‚Äî e.g. `sf code internal/common/grep/scan.go` emits
`struct,Options,"Root string; Patterns []string; Format string; ...;
MaxPerPattern int"` and `Run,"(opts Options, w io.Writer) error"` ‚Äî so the
entire task is answerable from structure alone, no bodies needed. (`../sofia`
left git-clean.)

## Step 3 ‚Äî pilot (1 rep, `sf` arm, nudge)

`sf` rep 1: `rc=0`, cost **$1.14**, **47 `sf code`** + 1 `sf grep`, 42 turns,
19,271 output tokens, **2 full-`Read` attempts (both hook-denied) ‚Üí 0 full

## `sf`-tool-usage verification: exercised structurally, across all 13 files

The load-bearing section. Verified the same way as t4 ‚Äî `calls.jsonl`
cross-checked against every transcript:

- **Calllog cross-check.** `sf`'s shared log (`~/.local/state/sofia/calls.jsonl`)
  tags each invocation with the worktree basename. `sf`-arm tags
  `sf-t5_dispatch-{1,2,3}` carry **131** entries (130 `code` + 1 `grep`):
  rep 1 = 47 `code` + 1 `grep`;
  rep 2 = 33 `code`; rep 3 = 50 `code`. **Zero** entries tagged to any `plain`
  worktree.
- **Transcript cross-check (tool_use enumerated per session):**
  - `sf` rep 1: Bash `sf code`√ó47 + `sf grep`√ó1; **2 `Read` attempts, both
    hook-denied ‚Üí 0 completed full Reads.**
  - `sf` rep 2: `sf code`√ó33; 2 `Read` attempts (denied); 0 full Reads.
  - `sf` rep 3: `sf code`√ó50; 2 `Read` attempts (denied); 0 full Reads.
  - `plain` rep 1: **16 `Read`** + 2 `Grep`, 0 `Bash`, 0 `sf`.
  - `plain` rep 2: **24 `Read`** + 3 `Grep`, 0 `Bash`, 0 `sf`.
  - `plain` rep 3: **24 `Read`** + 4 `Grep`, 0 `Bash`, 0 `sf`.
  Every `sf` rep went structural-first (`sf code <file>`, batching several per
  Bash call); every `plain` rep read the files in full. The nudge hook fired
  exactly twice per `sf` rep (the `permission_denials` in the run JSON) ‚Äî the
  agent hit it on its first full-`Read` reflex, then switched to `sf code`.
- **Not t1, not t4 ‚Äî confirmed empirically.** t1's failure was the agent
  grep-hunting and never touching `sf`; here `sf` did 130 structural reads and
  `plain` chose full `Read`s over grep (16‚Äì24 vs 2‚Äì4). t4's finding was `sf`
  drilling *one* file to death; here the 130 `sf code` calls spread across 13+
  files, and the answer genuinely needed all of them.

Net: this run compares "map 13 files structurally via `sf code`" against
"read those files in full" ‚Äî and on this task the structural path was cheaper
in dollars, because full Reads are the costly `cache_creation` that `sf` avoids.

## Verdict

**Yes ‚Äî on this task `sf` earned its tokens, and it's a genuine result inside
`sf`'s documented boundary.** Exercised structurally across 13 files (130
`sf code` calls, near-zero full Reads), `sf` came out **‚àí18.6% cheaper on
dollars** than an arm that read the same files in full ‚Äî the first win across
the three attempts, and the one the private `t1_deal` study predicted for
*many-file* comprehension. The mechanism is exactly as documented: `sf` does
**not** save tokens (it processed **+56%** more), it shifts volume out of
expensive full-Read `cache_creation` (**‚àí47%**) into cheap `cache_read`, and
at sonnet pricing that nets a dollar win. This is the opposite end of the same
boundary as t4: t4 forced `sf` onto a single file it needed in full and it lost
honestly; t5 gave `sf` many files it needed only the *shape* of, and it won
honestly.

Two things keep this a narrow, honest data point rather than a slam-dunk.
First, at N=3 the dollar win is fragile: `sf`'s cheaper two reps beat all of
plain, but its most-thorough rep cost more than all of plain, so the ‚àí18.6%
median hangs on which way the third rep fell. Second, the *stable* signal
(token volume) favours plain in every rep ‚Äî `sf`'s win is entirely a
token-*pricing* effect, and dollars are the noisy metric here (see
`micro.md`'s cache-warmup confound). `macro.md` is the relevant caution: on
sonnet it found the navigation win often *doesn't* materialise (the plan-phase
`sf` was +18% there; the ‚àí39% win showed only on opus). So read this as: on a
genuine many-file structural task, `sf`'s cost-shift *can* deliver a real
dollar win even on sonnet ‚Äî but it's model- and cache-sensitive and not
guaranteed.

## Caveats

- N=3/cell, one task, one model (sonnet), one operator ‚Äî a trend on a demo
  task, not a law. Consistent with `README.md`'s stated caveats.
- **The dollar win is a token-*pricing* win, not a token *saving*.** `sf` used
  +56% more `billed_in` in every rep; only because that extra volume is cheap
  `cache_read` (and it skips plain's expensive full-Read `cache_creation`)
  does it come out ahead in $. If cache pricing narrows, or on a run where the
  agent explores more (the pilot), the sign flips.
- **`sf`'s distribution straddles `plain`.** 2 of 3 `sf` reps beat every plain
  rep; 1 (the thorough pilot) lost to every plain rep. The median is a real
  but slender ‚àí18.6%.
- **Cost is noisy from cache warm-up** between sequential runs, and the runs
  were *not* order-counterbalanced (pilot sf/1, then plain/1‚Äì3, then sf/2‚Äì3).
  `billed_in` is the stable signal; it agrees on direction only for tokens
  (sf higher), not dollars.
- **Spend overran the ~$1‚Äì2 target: ~$5.3 on the six sessions** (sf
  $1.14/$0.75/$0.74, plain $0.84/$0.94/$0.93) plus judge. A genuine 13-file
  structural task is turn-heavy for *both* arms on sonnet (plain itself is
  ~$0.9/rep, ~4√ó t4's single-file plain), so REPS=3 both-arms cost several
  times t4. Reported transparently rather than trimmed mid-run.
- **Hook mode was `nudge` (production default), not t4's `strict`.** The
  strengthened structural-first preamble routed the agent to `sf code`; the
  nudge hook denied exactly the 2 full-`Read` reflexes per rep and the agent
  recovered in one turn each time. So this measures "preamble + armed nudge
  hook," the realistic production config.
- The task names the 13 files explicitly (both arms), so the measurement is
  "how each arm *reads* a fixed file set," not discovery cost ‚Äî deliberately,
  to isolate the structural-read-vs-full-read mechanism.
