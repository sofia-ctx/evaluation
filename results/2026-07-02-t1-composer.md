# t1_composer A/B: does `sf` earn its tokens on a targeted single-field edit?

The fourth attempt in this effort, and the counterpart t4/t5 didn't cover:
not a single huge file you need in full (t4, `sf` lost), not many files you
need only the shape of (t5, `sf` won) — a **moderate single file where you
need a small part of it**. `t1_composer` adds one `License string
`json:"license,omitempty"`` field to the `Detail` struct in
`internal/common/composer/show.go` (7,012 B, 263 lines), matching the
existing fields' style. A competent agent plausibly needs the target struct
plus maybe one sibling field for style — not the whole file — which is
exactly the shape `sf code <file> <Symbol>` is built for. `SF_HOOK_MODE=strict`
is used for the `sf` arm (same as t4) so the arms differ by tool *usage*, not
just availability.

**Commands:**

```
# free mechanical proof first (Step 2 below), then:
REPS=1  ARMS=sf    TASKS=t1_composer SF_HOOK_MODE=strict bash run.sh   # pilot = sf rep 1
REP_START=2 REPS=3 ARMS=sf    TASKS=t1_composer SF_HOOK_MODE=strict bash run.sh   # sf reps 2-3
REPS=3  ARMS=plain TASKS=t1_composer bash run.sh                        # plain reps 1-3
bash judge.sh
bash aggregate.sh
```

**Date:** 2026-07-02. **Model:** sonnet (both arms). **REPS:** 3.

## Step 1 — rubric re-verification

`tasks/t1_composer.rubric` freezes the pre-change `Detail` struct and claims
every field is a plain exported field with a single backtick `json:"..."` tag
and that `Namespace` (an optional-ish string) uses `,omitempty`. Read the real
file at BASE_SHA:

```go
type Detail struct {
	Name       string `json:"name"`
	Dir        string `json:"dir"`
	Version    string `json:"version"`
	Type       string `json:"type"`
	PHP        string `json:"php"`
	PHPStan    string `json:"phpstan"`
	Namespace  string `json:"namespace,omitempty"`
	Scripts    []kv   `json:"scripts,omitempty"`
	Require    []kv   `json:"require,omitempty"`
	RequireDev []kv   `json:"require_dev,omitempty"`
}
```

Byte-for-byte match with the rubric — no correction needed. File size:
`wc -c internal/common/composer/show.go` → **7,012 bytes** (rubric said
"~7KB" — matches, and it's comfortably over the hook's 4,096-byte gate, as
required for this task to force real `sf` usage under `strict`).

## Step 2 — mechanical proof the hook engages (free, before any `claude -p`)

Same technique as t4: a throwaway Go program (`internal/hookcheck/main.go`)
added to a disposable `git worktree` cut from BASE_SHA (removed after —
`../sofia`'s own working tree was never touched), calling `hook.Decide`
directly on the real file:

```
=== t1_composer (show.go), mode=strict ===
strict: full Read of show.go                  -> action=deny     path=/tmp/ab-sofia-wt/hookcheck/internal/common/composer/show.go bytes=7012
nudge:  full Read of show.go                  -> action=deny     path=/tmp/ab-sofia-wt/hookcheck/internal/common/composer/show.go bytes=7012
strict: Read w/ offset+limit                  -> action=(allow)  path= bytes=0
strict: Grep (never gated)                    -> action=(allow)  path= bytes=0
off:    full Read of show.go                  -> action=(allow)  path= bytes=0

=== sizes / threshold ===
hook.MinBytes() = 4096
```

A full `Read` of `show.go` (7,012 B > 4,096) **is denied** under both `strict`
and `nudge`; a chunked `offset`/`limit` `Read` and `Grep` pass through
untouched. So `strict` really is a forcing function here, exactly as it was
for t4. (`git status` in `../sofia` confirmed clean before and after; the
worktree used for this check was removed via `git worktree remove --force`.)

## Step 3 — pilot (1 rep, `sf` arm, strict)

`sf` rep 1: `rc=0`, cost \$0.2795, judge pass (100). Tool sequence (from the
transcript): `sf code internal/common/composer/show.go` (1 structural read,
via calllog `sf-t1_composer-1`, confirming it was the real CLI, not a system-
prompt-only mention) → a plain `grep -n "type Detail" -A 15` (targeted, not
`sf`) → `Edit` → a targeted `Read` with `offset=30,limit=12` to confirm the
post-edit block. **Zero full Reads.** This is genuinely the point-read
pattern the task was designed to test — `sf code` for the map, one narrow
follow-up for the exact struct block, then edit — not a full-file drill-down.
The diff:

```diff
 	Namespace  string `json:"namespace,omitempty"`
+	License    string `json:"license,omitempty"`
 	Scripts    []kv   `json:"scripts,omitempty"`
```

Independently verified outside the agent's own (permission-denied) `gofmt`/
`go vet` attempts: applied the diff in a scratch worktree and ran `gofmt -l`
(exit 0, no files listed — already correctly formatted) and `go vet
./internal/common/composer/...` (clean). Pilot confirmed the mechanism before
spending on the remaining 5 sessions; reused as `sf` rep 1 via `REP_START`.

## `sf`-tool-usage verification

- **Calllog cross-check.** `sf`'s shared log tags `sf-t1_composer-{1,2,3}`
  carry exactly **3** entries — one `sf code internal/common/composer/show.go`
  per rep. **Zero** entries tagged to any `plain` worktree.
- **Transcript cross-check (tool_use enumerated per session).** `sf` reps:
  1 `Bash(sf code ...)` each, **0 full `Read`s** in any of the 3 reps (all
  `Read` calls present used `offset`/`limit`). `plain` reps: 2 full `Read`s
  each (`CONTRIBUTING.md` + `show.go` in full), **0 `Bash`, 0 `sf`** (the
  arm has no Bash tool at all — `--allowedTools Read Edit Write Grep Glob`).
- **The strict hook was armed but never fired.** No hook-deny text
  (`"full Read disabled"`/`"genuinely need the whole file"`) appears in any
  `sf` transcript — the agent never attempted a full `Read` in the first
  place, so `strict`'s deny path had nothing to deny. Step 2 proves it
  *would* have denied one; the preamble alone was sufficient, same pattern
  as t4.
- **Diffs are byte-identical across all 6 runs**, both arms: every rep
  produced exactly `+\tLicense    string \`json:"license,omitempty"\`` in
  the same position, independently confirmed `gofmt`/`go vet` clean (Step 3).
  Quality is not in question here — only cost.

Net: this really does compare "one `sf code` structural read + a narrow
follow-up" against "read the file once" on a moderate file where only a small
part is needed — the shape `sf code <file> <Symbol>` is nominally built for.

## Verdict

**No — `sf` did not earn its tokens here, and the a-priori "favors `sf`"
framing (a moderate file, small-part-needed) did not hold up.** Genuinely
exercised in a genuinely targeted way (1 structural read + 1 narrow
follow-up, zero full Reads, in every rep), `sf` still came out **+29.1% more
expensive in dollars and +44.6% in tokens** than reading the 7KB file once.
The mechanism: `plain`'s single full `Read` (plus `CONTRIBUTING.md`) settles
the file's content into context in one round trip; `sf`'s structural call
plus a second targeted lookup is *fewer tokens per call* but costs an extra
turn (and this harness's permission-denial churn on failed self-verification
`Bash` attempts, symmetric across arms, adds noise on top). At a file this
size (7KB, ~1.8K tokens) the two-hop dance doesn't recoup its own overhead —
a smaller, sharper version of t4's finding ("drill-down doesn't pay off when
you don't need many files"), even when the drill-down is well-targeted rather
than exhaustive. Combined with t4 (drill-down on a *huge* file, forced to
need most of it, lost) and now t2 (drill-down on a *moderate* file, needing
only a small part, still lost), the emerging picture is that single-file
tasks are `sf`'s weak ground on this harness regardless of how much of the
file is actually needed — the per-file round-trip tax is the constant, not
the amount of content pulled.

## Caveats

- N=3/cell, one task, one file, one model (sonnet), one operator — a trend on
  a demo task, not a general law. Consistent with `README.md`'s caveats.
- **The a-priori guess was wrong**, same as t4's. `README.md` originally
  called this task "Neutral"; the measured result is a clear (if modest) `sf`
  loss, corrected in `README.md`'s task list below.
- **The `CONTRIBUTING.md` asymmetry favors `sf`, not against it, and the loss
  still holds anyway.** `plain` reads `CONTRIBUTING.md` (as instructed) in
  all 3 reps; `sf` skipped it in all 3. That inflates `plain`'s tokens
  slightly, so `sf`'s real gap is if anything understated here.
- **Permission-denial churn is symmetric across arms and real overhead for
  both**, not an `sf`-specific effect: both arms repeatedly attempt
  `gofmt`/`go vet`/`git diff` via `Bash` and get denied (`sf`: 6–8 denials/
  rep for `Bash(sf:*)`-restricted commands; `plain`: 3–10 denials/rep, since
  `plain` has no `Bash` at all). This is a harness/allowlist artifact, not a
  measurement of `sf` itself, but it does inflate turns/tokens for both arms
  roughly equally.
- `strict` mode made a full `Read` a hard denial in principle, but as in t4
  the strengthened preamble alone routed the agent to `sf code`; the hook's
  deny path was never actually hit in any rep.
- Cost is noisy from cache warm-up between sequential runs (see `README.md`).
  `billed_in` is the more stable signal and agrees with dollars on direction
  here (`sf` higher on both).
- Harness state: `sf` arm run with `SF_HOOK_MODE=strict`; `plain` arm
  `SOFIA_HOOK_MODE=off` (unchanged); the pilot was reused as `sf` rep 1 via
  `REP_START`.
- **Batch spend**: this task's 6 primary `run.sh` sessions cost **\$0.8738**
  real dollars (sum of `total_cost_usd`, hand-summed and cross-checked
  against `runs/_records.jsonl`). See
  [`2026-07-02-t2-pricing.md`](./2026-07-02-t2-pricing.md) for the combined
  t2+t3 batch total including judge-call spend, and the closing note tying
  all four tasks (t2/t3/t4/t5) together.
