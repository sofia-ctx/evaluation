# t1_calllog A/B: does `sf` earn its tokens on multi-file comprehension?

A single-task run of this repo's harness against `sofia-ctx/sofia`, targeting
the task shape (`t1_calllog`) that prior private measurement
([`micro.md`](https://github.com/sofia-ctx/sofia/blob/main/docs/measurements/evaluation/micro.md)'s
analogous `t1_deal`) found favored `sf` — tracing a pattern across many
files. Chosen for that reason, not cherry-picked after the fact. The number
below is what actually came out, including where it disagrees with that
prior result.

**Command:**

```
REPS=3 ARMS="sf plain" TASKS=t1_calllog MODEL=sonnet bash run.sh
bash judge.sh
bash aggregate.sh
```

**Date:** 2026-07-02. **Model:** sonnet (both arms). **REPS:** 3.
**BASE_SHA:** `257718bfc4d6fee74322c24f4c90e8db02c99efa` (sofia-ctx/sofia,
2026-07-02 09:06:52 +0500) — confirmed an ancestor of the `../sofia` clone's
HEAD (`35c34ee`) at run time.

## TL;DR

1. **`sf` did not win here.** Both cost and token volume are *higher* on
   the `sf` arm — the opposite of the direction this task was chosen to
   test, and the opposite of what `micro.md`'s equivalent task (`t1_deal`)
   found (there, `sf` was +91% tokens but −17% *dollars*, from shifting
   volume into cheap `cache_read`). Here `sf` is worse on every primary and
   secondary metric: cost, tokens, `cache_read`, turns, and wall time.
2. **The reason isn't subtle: the `sf` CLI was never actually invoked.**
   Across all 3 `sf`-arm reps, zero `sf ...` commands appear in the
   transcripts, and zero matching entries exist in the shared calllog. The
   agent had `sf` reachable (on `PATH`, `Bash(sf:*)` allowed, told to
   prefer it in the system prompt) and simply used the built-in `Grep`/
   `Read` tools for the whole task instead. This run is not a measurement
   of "`sf` vs `plain`" so much as "an agent that chose not to pick up the
   tool it was handed" — see the verification section below.
3. **Quality is equal.** All 6 runs passed the judge, scores 97–100 on
   both arms; no files were modified in any run (empty diffs), as the task
   required.

## `sf`-tool-usage verification: `sf` was not exercised

The whole point of the `sf` arm is that the agent actually calls `sf`'s
tools. It didn't, in any of the 3 reps:

- **Calllog cross-check.** `sf`'s own shared log
  (`~/.local/state/sofia/calls.jsonl`, resolution order in
  `../sofia`'s `internal/calllog/calllog.go`) tags every real invocation
  with the project directory's basename — for these worktrees, tags like
  `sf-t1_calllog-1`. `grep -c "t1_calllog" ~/.local/state/sofia/calls.jsonl`
  returns **0** for the entire run window (09:27–09:40 UTC on 2026-07-02).
  No `code`, `grep`, `changed`, or `hook.nudge` entries were logged for any
  of the three `sf`-arm sessions.
- **Transcript cross-check.** Each `sf`-arm run's session transcript
  (`~/.claude/projects/-tmp-ab-sofia-wt-sf-t1-calllog-{1,2,3}/<session_id>.jsonl`,
  located via the `session_id` in `runs/sf/t1_calllog/*.json`) was read
  directly and every `tool_use` block enumerated:
  - rep 1: 3 `Bash`, 9 `Grep`, 1 `Read` — the 3 `Bash` calls were
    `sed -n ... CONTRIBUTING.md` and two `for f in ...; do grep ...; done`
    loops (the second and third of which were denied by the
    `--allowedTools "Bash(sf:*)"` gate — visible in
    `runs/sf/t1_calllog/1.json`'s `permission_denials`), after which the
    agent fell back to individual `Grep` tool calls rather than `sf grep`.
  - rep 2: 2 `Bash`, 10 `Grep` — same pattern, no `sf`.
  - rep 3: 1 `Bash` (`grep -rl 'calllog\.Start('...`), 5 `Grep`, 1 `Read` —
    same pattern, no `sf`.
  - None of the three transcripts contain the string `sf code`, `sf grep`,
    or `sf changed` anywhere.
- **The PreToolUse nudge hook also never fired.** `sf hook pre` only
  writes a calllog entry (tool `hook.nudge`) when it actually denies or
  suggests — silent pass-through otherwise (`internal/common/hook/cmd.go`).
  Since no such entries exist for these sessions, the hook's threshold
  (default ≥4096 bytes on a full `Read`/`cat`) was never crossed — the
  agent's one `Read` per rep and its many small `Grep` calls apparently
  stayed under it, so the nudge that's supposed to redirect toward `sf`
  never had a chance to trigger.

Net: this run measured "`Grep`+`Read` told to prefer `sf`" against
"`Grep`+`Read` told not to use `sf`" — both arms ended up doing the same
kind of work, and the `sf` arm's extra cost/tokens/turns are the cost of
system-prompt overhead (the longer `sf`-preamble, `sf --help` /
`CONTRIBUTING.md` framing) plus one rep (rep 1) that ran noticeably more
turns and larger `Read`, not `sf` itself.

## Verdict

**No, `sf` did not win on this run** — not by a little, by a clear margin
in the wrong direction (+30% cost, +57% tokens, +86% turns), on the exact
task shape it was expected to favor. But this result doesn't actually
speak to `sf`'s value, because `sf` was never invoked in any of the 3
`sf`-arm reps: the preamble told the agent to prefer it, the hook was
registered and live, and the agent still solved the whole task with
`Grep`/`Read` alone. What this run *does* show is that the nudges
(system-prompt instruction + PreToolUse hook) are not reliable enough to
guarantee `sf` gets picked up even on a task explicitly shaped for it —
consistent with `micro.md`'s own finding #2, that the instructions/hook
tuning still needs work. With N=3 reps, one task, one model (sonnet), one
operator, and (per this run) effectively one arm actually differentiated
by tool availability rather than tool *usage*, this is not a general
verdict on `sf` — it's a specific, honest data point: on this occasion,
with this agent's choices, `sf`'s presence on `PATH` cost more and helped
not at all, because it went unused. See
[`micro.md`](https://github.com/sofia-ctx/sofia/blob/main/docs/measurements/evaluation/micro.md)
and
[`macro.md`](https://github.com/sofia-ctx/sofia/blob/main/docs/measurements/evaluation/macro.md)
in `sofia-ctx/sofia` for the larger, private-codebase study this task
selection is grounded in, including runs where `sf` *was* exercised and
did show a dollar-cost win on comprehension tasks.

## Caveats

- N=3/cell, one task, one model (sonnet), one operator — consistent with
  this harness's own stated caveats (see `README.md`), this is a trend on
  a demo task, not a general law about `sf`.
- The dominant finding here isn't cost or tokens, it's tool-adoption: an
  agent given `sf` and told to prefer it chose not to use it at all in 3/3
  reps. Re-running with a stronger nudge (lower `SOFIA_HOOK_MIN_BYTES`,
  or `strict` hook mode) would be a different, useful experiment, but is
  out of scope for this run per the task's fixed scope.
- Cost is noisy from cache warm-up between sequential runs (see
  `micro.md`'s confound writeup); `billed_in` is the more stable signal,
  and it agrees with the dollar figure's direction here.
- Aside, not affecting the arms comparison: the task's own frozen
  `.rubric` states "22 files" / "15 under `internal/common/`" as summary
  counts, but its own enumerated file list contains 23 files (16 under
  `internal/common/`) — an internal arithmetic slip in the rubric text,
  caught independently by the judge in all 6 runs, which scored against
  the (correct) enumerated list rather than the (incorrect) summary count.
  Both arms were judged identically and fairly by the same rubric, so it
  doesn't bias the `sf`-vs-`plain` comparison, but it's worth fixing in
  `tasks/t1_calllog.rubric` before this task is reused.
