# t3_pricing A/B: does `sf` avoid forcing itself onto a trivially small file?

The counterpart to t1's deleted invalid attempt, on the opposite question:
not "does the agent fail to adopt `sf` when it should," but **does the agent
correctly avoid forcing `sf` when a file is trivially small and the `sf`-arm's
own preamble says a plain `Read` is fine?** `t3_pricing` adds one
`"claude-haiku-5": cachePrice(1.20, 6.00)` entry to the `modelPrices` map in
`internal/cc/pricing.go` (68 lines, 2,918 bytes — comfortably under the
hook's 4,096-byte gate). Run with the harness's **default hook mode**
(`SF_HOOK_MODE` unset → `nudge`), not t2's forcing `strict` — the point is to
see what the agent does when nothing is forcing it either way.

**Commands:**

```
# free mechanical proof first (Step 2 below), then:
REPS=1  ARMS=sf    TASKS=t3_pricing bash run.sh                 # pilot = sf rep 1
REP_START=2 REPS=3 ARMS=sf    TASKS=t3_pricing bash run.sh      # sf reps 2-3
REPS=3  ARMS=plain TASKS=t3_pricing bash run.sh                  # plain reps 1-3
bash judge.sh
bash aggregate.sh
```

**Date:** 2026-07-02. **Model:** sonnet (both arms). **REPS:** 3.
**BASE_SHA:** `257718bfc4d6fee74322c24f4c90e8db02c99efa` (sofia-ctx/sofia),
working tree byte-identical to BASE_SHA for the target file (see t2's report
for the shared verification command).

## Step 1 — rubric re-verification

`tasks/t3_pricing.rubric` claims: `modelPrices` is `map[string]ModelPrice`,
every entry is `"<id>": cachePrice(input, output)`, no entry hand-builds a
`ModelPrice{...}` literal except inside `cachePrice` itself and the
zero-value in `priceFor`, `claude-haiku-5` doesn't exist yet, and the closest
existing key is `claude-haiku-4-5` at 1.00/5.00. Read the real file at
BASE_SHA:

```go
var modelPrices = map[string]ModelPrice{
	"claude-opus-4-8":   cachePrice(5.00, 25.00),
	...
	"claude-haiku-4-5":  cachePrice(1.00, 5.00),
}
```

All confirmed — `claude-haiku-4-5` is the last entry at 1.00/5.00,
`claude-haiku-5` is absent, and every entry (including this one) calls
`cachePrice`. File size: `wc -c internal/cc/pricing.go` → **2,918 bytes**,
`wc -l` → **68 lines** — exactly matching the rubric, and well under 4,096.
No correction needed.

## Step 2 — mechanical proof the hook is a non-factor here (free)

Same throwaway `hook.Decide` program as t2 (Step 2), same disposable
worktree, this time on `pricing.go` across every mode:

```
=== t3_pricing (pricing.go), mode=nudge (default) ===
nudge:  full Read of pricing.go               -> action=(allow)  path= bytes=0
strict: full Read of pricing.go               -> action=(allow)  path= bytes=0
suggest: full Read of pricing.go              -> action=(allow)  path= bytes=0

=== sizes / threshold ===
hook.MinBytes() = 4096
```

Confirmed: a full `Read` of `pricing.go` (2,918 B < 4,096) is **allowed**
under every mode — `nudge`, `strict`, and `suggest` all pass it through
unmodified, because `hook.Decide` returns early (`fi.Size() < minBytes`)
before the mode switch is even consulted. The hook is a non-factor here *by
design*, not by luck — confirmed mechanically, not assumed.

## Step 3 — pilot (1 rep, `sf` arm, default `nudge`)

`sf` rep 1: `rc=0`, cost \$0.4530, judge pass (100). Tool sequence: attempted
`Skill(claude-api)` (denied — not in `allowedTools`; a global skill-trigger
side effect of the task text naming a model id, see Caveats) → read
`CONTRIBUTING.md` → `sf code internal/cc/pricing.go` (structural) →
`Grep` → a **full `Read`** of the whole 68-line file → `Edit` → several
`gofmt`/`go build`/`go vet` attempts (all denied, `Bash` restricted to
`Bash(sf:*)`) → two more targeted `Read`s to self-verify since the shell
checks were blocked.

**This is the honest finding, looked at without a preferred outcome going
in: the arms did not converge.** The task's own preamble says "for a single
small file you already need in full, one Read is fine — don't force a
structural-read-then-point-read dance where it doesn't pay for itself," and
the hook mechanically cannot intervene either way (Step 2). Despite that, the
agent still reached for `sf code` (and a `Grep`) *before* falling back to
the full `Read` it needed anyway — doing both, not one or the other. That is
exactly the "pointless overhead" failure mode the task was designed to be
able to detect, and it happened on the very first (unforced) rep.

## `sf`-tool-usage verification

- **Calllog cross-check.** `sf`'s shared log tags `sf-t3_pricing-{1,2,3}`
  carry **7** `code` entries — rep 1: 3×, rep 2: 2×, rep 3: 2×. **Zero**
  entries tagged to any `plain` worktree.
- **Transcript cross-check.** Bash command lines actually run per rep:
  - `sf` rep 1: `sf code pricing.go` (1×), then repeated `gofmt`/`go build`/
    `go vet` attempts (all denied). **2 full `Read`s** completed anyway.
  - `sf` rep 2: `sf code pricing.go` → `sf code pricing.go modelPrices`
    (structural, then a single targeted symbol body — the well-calibrated
    point-read the task's shape rewards) → `gofmt`/`go build` attempts
    (denied). **0 full Reads** — the only `sf` rep that stuck to the
    structural+point-read path without also falling back to a full Read.
  - `sf` rep 3: same `sf code` → `sf code ... modelPrices` pattern, but
    **1 full `Read`** completed anyway on top of it.
  - `plain` reps 1–3: no `sf`, no hook (`SOFIA_HOOK_MODE=off`); 2–4 full
    `Read`s each (`CONTRIBUTING.md` + `pricing.go`, occasionally re-read
    after edit); repeated `gofmt`/`go build`/`go vet`/`go test` attempts, all
    denied (`plain` has no `Bash` tool at all).
  - **No hook-deny text** (`"cheaper next time"`/`"full Read disabled"`/
    `"genuinely need the whole file"`) appears in any `sf` transcript,
    confirming Step 2's prediction live: the hook never had anything to
    engage with, because the file never crossed its size gate.
- **The finding, stated plainly:** in 2 of 3 `sf` reps (1 and 3), the agent
  used `sf code` *and* a full `Read` on the same 68-line file — genuine
  over-application, not hook-forcing (the hook was mechanically inert the
  whole time). Only rep 2 followed the calibrated point-read pattern the
  preamble asks for. The arms did not converge; `sf`'s own guidance
  ("one Read is fine ... don't force a structural-read-then-point-read
  dance") was followed inconsistently even though nothing was stopping the
  agent from just reading the file.

## Verdict

**No — `sf` did not earn its tokens, and unlike t2 this is squarely a
finding about over-application rather than about the drill-down mechanism
itself.** The hook was mechanically proven inert at this file size (Step 2),
confirmed inert in every live run (zero deny-text hits), and the task's own
preamble explicitly told the `sf` arm a plain `Read` is fine here — and the
agent still reached for `sf code` (occasionally plus a targeted symbol body)
before or alongside a full `Read` in every rep. The result: **+9.0% cost,
+29.2% tokens**, a real but modest loss — smaller than t2's because the
extra `sf code` call is cheap in absolute terms on a 68-line file, but not
zero, and not converging with `plain` as the calibrated-preamble hypothesis
hoped. One rep (sf/2) shows the preamble *can* produce the intended
calibrated behavior (structural + single point-read, no full Read at all,
and it was the cheapest `sf` rep) — so this isn't "the preamble never
works," it's "the preamble doesn't reliably override the reflex to reach for
the fancier tool first," even on a file explicitly called out as too small
to bother.

## Closing note: the 2×2 boundary map, complete

Four tasks now cover the cross of *how much of a file you need* × *how big
the file is*:

| | need most/all of the file | need only a small part |
|---|---|---|
| **large file** (t4, 12KB) | `sf` **lost**: +8.6% $, +94% tokens. Drilling one file symbol-by-symbol reconstructs almost the whole file across more round trips than one `Read`. | *(not tested — `t5` covers the many-file analogue instead)* |
| **moderate file** (t2, 7KB) | *(not this task's shape)* | `sf` **lost**: +29.1% $, +45% tokens, even with a genuinely targeted structural+point-read usage pattern. |
| **small file** (t3, 2.9KB, below the hook's gate) | `sf` **lost**: +9.0% $, +29% tokens — the hook can't force anything this small, but the agent forced `sf` on itself anyway (over-application). | |
| **many files** (t5, 13 files) | *(not this task's shape — the many-file question is "need the shape, not the bodies")* | `sf` **won**: −18.6% $ (though +56% tokens — a cache-pricing shift, not a token saving), fragile at N=3. |

The pattern that survives all four: **`sf` needs *many files* to pay off, not
just "a smaller amount of one file."** Every single-file task tried here —
huge (t4), moderate (t2), or trivially small (t3) — lost on both dollars and
tokens, regardless of whether the drill-down was forced (t2/t4, `strict`) or
happened voluntarily against the tool's own advice (t3, `nudge`, hook
inert). The per-file round-trip tax (`sf code` for structure, then one or
more follow-ups) is roughly constant per file touched; it only nets positive
once there are enough files that `plain`'s alternative — full-Reading each
one — accumulates more expensive `cache_creation` than `sf`'s many cheap
`cache_read`-backed structural calls can offset (t5's mechanism exactly).
t3 additionally shows the failure mode isn't purely mechanical: even with the
hook proven inert and the preamble explicit, the agent reached for the
structural tool out of habit on 2 of 3 reps — a calibration problem in the
prompt/skill guidance, not a limitation of the hook.

## Caveats

- N=3/cell, one task, one file, one model (sonnet), one operator — a trend on
  a demo task, not a general law. Consistent with `README.md`'s caveats.
- **A confound shared by both arms, not `sf`-specific:** the task text names
  a model id (`claude-haiku-5`), which matches the global `claude-api`
  skill's trigger heuristic ("names a Claude/Anthropic model"). Every rep in
  *both* arms attempted `Skill(claude-api)` once or twice and was denied
  (not in `allowedTools`) — a wasted turn symmetric across arms, not a bias
  toward either.
- **Permission-denial churn is symmetric and real overhead for both arms**,
  same as t2: repeated `gofmt`/`go build`/`go vet`/`git diff`/`go test`
  attempts via `Bash`, mostly denied (5–7 denials/rep for `sf`, 7–10/rep for
  `plain`, since `plain` has no `Bash` at all). Harness/allowlist artifact,
  not a measurement of `sf` itself.
- Cost is noisy from cache warm-up between sequential runs (see `README.md`).
  `billed_in` (the more stable signal) agrees with dollars on direction here.
- Harness state: `sf` arm run with `SOFIA_HOOK_MODE` unset (→ `nudge`,
  production default); `plain` arm `SOFIA_HOOK_MODE=off`. The pilot was
  reused as `sf` rep 1 via `REP_START`.

## Batch spend (t2 + t3 combined) — a real, reported overrun

The two tasks were pitched at **\$2–4 total**. The real total ran well past
that:

- **12 primary `run.sh` sessions** (t2: 3×`sf` + 3×`plain`; t3: 3×`sf` +
  3×`plain`): **\$3.8925** exactly (`total_cost_usd`, hand-summed and
  cross-checked against `runs/_records.jsonl`: t2 \$0.8738 + t3 \$2.1878).
  This is the number comparable to t4's \$1.19-for-6-sessions and t5's
  \$5.3-for-6-sessions.
- **Judge-call spend, measured, not assumed.** `judge.sh`'s per-run judge
  call (`claude -p`, `--allowedTools ''`) is documented as "cheap" — it isn't,
  in this environment. It inherits the full ambient skill-catalog/CLAUDE.md
  system prompt (visible verbatim in the judge transcripts, same
  `skill_listing` attachment seen in every agent session) and, per the
  transcripts, the judge model frequently produces several thousand output
  tokens despite the "return ONLY a single-line JSON object" instruction —
  neither of which the harness's cost model accounts for. Reconstructed from
  the 18 completed judge-call transcripts in this session's window (usage
  tokens × sonnet's own published rates — the same `cachePrice(3.00, 15.00)`
  this batch's `t3_pricing` task edits): **≈\$3.94**, individual calls
  ranging \$0.06–\$0.35. One call's reconstructed estimate (\$0.212) was
  cross-checked against its actual reported `total_cost_usd` (\$0.2269) —
  within ~7%, validating the method. Two additional judge attempts were
  killed mid-flight by a tooling timeout on my end (redundant reruns caused
  by re-judging already-completed reps) and never produced a billed
  response, so they're not counted.
- **Total real spend, this batch: ≈\$7.83** (\$3.8925 primary + ≈\$3.94
  judge) — roughly double the \$2–4 estimate, and proportionally worse than
  `t5`'s previously self-reported overrun to ~\$5.34 on a single task.
- **Why the estimate missed:** both pilots already ran \$0.28 (t2) and
  \$0.45 (t3) — higher than "simpler/smaller-file" suggested — and 3 reps ×
  2 arms × 2 tasks scales that linearly before judge cost is even
  considered. The bigger miss was structural: `judge.sh`'s cost was assumed
  negligible and isn't measured anywhere in this repo's prior reports
  (`t4`/`t5` also ran the same judge calls and likely paid a similar,
  uncounted premium — this isn't unique to this batch, just the first time
  it's been measured and reported). Flagged here transparently rather than
  retroactively trimmed.
